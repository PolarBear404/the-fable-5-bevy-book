写第 $ARGUMENTS 章（下文章号记作 XX）。严格按以下顺序执行，前一步没完成不进入下一步。

**分工纪律**：定范围、写代码、写正文是本章的心智主链，必须在当前会话亲手做——正文作者
必须亲眼见过每一次编译报错与实测输出（先跑后写）。插图、WASM demo、审阅是噪音型阶段，
一律用 Agent 工具派发子 agent，凭文件交接（工单 + 规范文档），不依赖本会话上下文。

0. **开工单**：创建 `workorders/chXX.md`，分节：范围与衔接／示例设计（场景、角色名单、
   各 listing 教学意图）／实测台账／坑与报错素材／插图规格清单／demo 判定／收尾清单。
   此后每步的决策与实测**随手逐条写入工单**——上下文被压缩后凭它恢复现场，子 agent 凭
   它接活，收尾时它是 PROGRESS.md 行的原料。工单随章入库。

1. **定范围**：读 OUTLINE.md 本章条目与 PROGRESS.md；浏览前一章正文结尾与后一章大纲，
   确保衔接——不依赖未讲概念、不重复已讲内容。按 CLAUDE.md「交互演示规范」判定本章
   是否配内嵌 demo（默认不配，从严）。发现合理范围与大纲有出入时先停下向用户说明，
   不擅自改大纲。结论写入工单。

2. **代码先行**：在 code/ 下创建本章 crate（命名如 ch23-gltf，加入 workspace 的
   members）。写出本章全部示例代码。铁律：每个 Bevy API 先在 vendor/bevy/ 确认存在
   与签名；官方 examples 是选材第一参考。
   - **微实验加密**：每个核心概念至少配一个「拨一下」实验——改一个值或开关，亲眼对比
     差异（范本：ch22 的 512 贴图对比、级联收紧）。实验优先做成 listing 变体或运行时
     按键档位，让读者能在正文里跟着拨。
   - 示例设计守质量标尺：系统是角色、输出是台词，零填充行、零空帧；每章保留至少一处
     「亲眼看报错」——真实报错做成可运行（或编译失败放 no-compile/）的 listing，
     不是一句话带过。
   - 逐个 `cargo check -p <crate>` 全绿；可运行示例逐个 `cargo run` 实跑，**输出逐字
     记入工单实测台账**——台账是正文一切输出块与行为断言的唯一来源。

3. **写正文**：动笔前重读第 1 章找语感（风格基准）。在 book/src/ 写本章 .md（文件名
   英文 kebab-case）。所有 Rust/TOML 代码块一律 `{{#include}}` 自 code/（ANCHOR 截取），
   禁止手写。两条硬规（试行标尺：ch23 验收通过后删除此括号并转正进 CLAUDE.md）：
   - **逐参数讲解**：listing 里每个首次出现的 API/类型/字段都要交代；非显然的取值必须
     给理由（为什么 30 米、为什么 2048）。不允许「代码自解释」式跳过。
   - **微实验入文**：每个概念的实验以「问题→实验→结论」弧线呈现，配真实输出或插图；
     纯 API 罗列式小节是缺陷。
   篇幅校准：全章预期 45k 字符量级（现行章节约 1.5×）——**篇幅只准从上述两类内容长
   出来**，不准用过场话、重复、比喻堆砌凑数；写完明显低于量级时，先自查漏讲的参数和
   漏做的实验，而不是硬拉长。机制解说按需给：当它改变读者的用法或心智模型时必须给
   （配手绘 SVG，范本 22.6「影子是怎么算出来的」），不设每章配额。
   插图打桩：正文图位直接写详尽中文 alt 文本 + caption——alt 即插图规格，同步登记
   工单「插图规格清单」（编号、静/动、画面内容、所属小节）。

4. **插图（派发）**：用 Agent 工具派发子 agent，提示词须令其先读 CLAUDE.md「插图规范」、
   `scripts/figures-ops.md` 和本章工单的插图规格清单，产出 `scripts/make_chXX_figures.py`
   与全部图，逐图对照规格回报。需要合成美术/音频资产（`make_chXX_assets.py`，纯 Python
   原则）可一并派发，但画什么、什么风格由主会话先写进工单。收货后主会话亲自 Read 每张
   图核对图文一致——不符时修代码或修正文，不准 P 图。

5. **WASM demo（仅判定适用的章）**：派发子 agent，先读 CLAUDE.md「交互演示规范」与
   `scripts/wasm-ops.md`，产出 `scripts/build_chXX_wasm.py`、宿主页与正文嵌入段，并在
   preview 里实跑验证、回报证据（AdapterInfo 日志、全部交互路径的验证记录）。主会话
   复核证据后才算过。

6. **构建验证**：跑 `scripts/fix_quotes.py` 清直引号；`mdbook build book` 通过、无
   include 解析失败；检查本章内部链接、Listing/Figure 编号连续；扫构建产物中的字面
   `**`（粗体以中文标点收尾的 CommonMark 坑，toc.html 的 JS 除外）。

7. **新鲜眼审阅（派发）**：派发只读子 agent 审本章，只报告不修改：①技术准确性——对
   vendor/bevy 抽核 API 签名与行为断言；②两条硬规逐节达成情况；③体例对 ch01 与
   CLAUDE.md 行文规范（含无痕原则）；④图文一致抽查。主会话逐条处置——修掉，或在工单
   记下不修的理由；有重大修改则再派一轮复核。

8. **收尾**：更新 book/src/SUMMARY.md 与 PROGRESS.md（工单是原料）；git commit（连同
   工单），信息格式 `ch23: 章题`。向用户汇报：本章覆盖的大纲点、有意留到后续章的点、
   与前后章的衔接处，以及标尺自评——两条硬规的达成情况与最终字符量。
