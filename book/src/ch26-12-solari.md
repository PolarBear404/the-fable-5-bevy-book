# 一瞥未来：bevy_solari 实验性光追

收工前，盛师傅聊起一件行业新鲜事。回望这两部 3D 的篇章，你会发现光栅化渲染像一间挂满补丁的老铺子：影子不是光挡出来的，是**阴影贴图**从灯的视角偷拍的（22.6）；镜面反射不是光弹回来的，是**反射探针**提前拍好的环境照（22.11）；缝隙里的暗不是光到不了，是**接触阴影**在屏幕空间摸出来的（22.7）——每一项都是聪明的近似，每一项也都有自己那节里写过的翻车姿势。补丁越打越多，是因为光栅化从第一天起就绕开了那个最贵的问题：**光到底走了哪条路**。

**光线追踪**（ray tracing）掉头直面它：从相机（或光源）射出真实的光线，与场景求交、反弹、累积——影子、反射、间接光照全是同一套物理的自然结果，不再各配一个补丁。代价同样直白：算力吞金，直到近几年的专用硬件（RT core）才把它拖进实时的门槛。

Bevy 的回应是 **`bevy_solari`**——官方的实验性光追 crate，眼下的形态：

- **`SolariPlugins`** 插件组，装两件：`RaytracingScenePlugin`（把网格组织进光追加速结构，行话叫 BLAS）和 `SolariLightingPlugin`（实时的光追直接光 + 间接光）；
- 用法的形状你已经很眼熟：相机挂 **`SolariLighting`** 组件，参与光追的网格在 `Mesh3d` 之外再挂一个 **`RaytracingMesh3d`**（还是“组件即开关”的老语法，只是这次要给每件参与的货逐个发牌）；
- 硬件门槛写在 `SolariPlugins::required_wgpu_features()` 里：`EXPERIMENTAL_RAY_QUERY` 打头的一串 wgpu 特性——翻译过来是“要一块支持光线查询的显卡和对应驱动”，拿不到这些特性的机器直接开不了工；
- crate 藏在**非默认** feature `bevy_solari` 后面，连官方示例都要 `--features bevy_solari` 才编译——“实验性”三个字是认真的：API 会变、平台受限、性能仍在打磨；
- 附带一个不进默认插件组的 **`pathtracer`** 模块：非实时的路径追踪器，官方拿它当“标准答案机”——同一场景离线慢慢算一张物理正确的图，校验实时管线的偏差。

本书对它的态度也写明白：**只看地图，不上车**。它的 API 尚未定型，本节不给可运行的示例，也不描述实测行为；等它毕业转正，值得一整章。想先睹为快的读者，官方仓库的 `3d/solari` 示例是入口（注意它的 required-features 清单），跑不跑得起来取决于你的显卡答不答应。

对读者更有价值的是**坐标系**：本章学的一切——辉光、景深、tonemapping、TAA——在光追时代照样在岗，因为它们处理的是“拿到光照结果之后怎么呈现”；会被光追逐步替换的是“光照结果怎么算出来”的那半边（阴影贴图、探针、屏幕空间技巧）。把补丁铺子看成过渡形态，你对渲染管线的理解就有了纵深。
