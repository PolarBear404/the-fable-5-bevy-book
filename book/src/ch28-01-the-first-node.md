{{#include ../../code/ch28-ui-layout/examples/listing-28-01.rs:setup}}

<span class="caption">Listing 28-1：第一块牌子——一个 `Node`、一块背景色，就是最小的 UI（examples/listing-28-01.rs）</span>

先看这坨 spawn 里**没有**什么：没有 `Transform`，没有 `Sprite`，没有任何“坐标”。第 12 章以来我们摆放东西的全部手艺，在 UI 世界里一概不用——`Node` 的位置和尺寸由**布局系统**说了算，你只负责填意图：

- **`width` / `height`**——这块牌子想要多宽多高。类型是 `Val`（下下节的主角），`px(320)` 是 `Val::Px(320.0)` 的便捷写法——320 个逻辑像素。`Node` 有四十多个字段，全部给了默认值，所以永远是“填几个意图、剩下 `..default()`”的写法；
- **`BackgroundColor`**——牌面的底色。单独一个组件而不是 `Node` 的字段，默认值是 `Color::NONE`（完全透明）——不挂它也能布局，只是看不见。

跑起来：

```console
cargo run -p ch28-ui-layout --example listing-28-01
```

牌子出现在**左上角**。这不是巧合，是 UI 世界的两条地界规矩：

- 没有父节点的 `Node` 是**根节点**，它的地盘就是整个视口；
- UI 的坐标原点在**左上角，Y 轴朝下**——跟第 12 章“Y 朝上”的世界坐标正好相反。这是从网页布局沿袭的约定：界面元素从上往下、从左往右排，跟读文章一个方向。

## 一窝组件

第 3 章讲过 required components：spawn 一个组件，它声明依赖的组件自动跟着上。`Node` 是全书至今拖家带口最多的一个——它直接声明的随从就有十二位：

| 自动带上的组件 | 干什么的 |
|---|---|
| `ComputedNode` | 布局结算后的实测账本：尺寸、缩放……（下一节主角） |
| `ComputedStackIndex` | 结算后的叠放次序（28.8） |
| `ContentSize` | 内容自带的尺寸——文本、图片靠它向布局报体格（28.10、28.11） |
| `ComputedUiTargetCamera` | 结算后归哪台相机管（28.12） |
| `ComputedUiRenderTargetInfo` | 结算后目标画布的尺寸与缩放（28.12） |
| `UiTransform` | UI 专用的平移/旋转/缩放——布局之后再做视觉微调，本章不展开 |
| `BackgroundColor` | 底色，默认全透明 |
| `BorderColor` | 边框色，四条边各管各（28.4） |
| `FocusPolicy` | 拦不拦截交互焦点（第 29 章） |
| `ScrollPosition` | 滚动偏移（第 29 章） |
| `Visibility` | 第 12 章的老相识：画不画 |
| `ZIndex` | 同级之间谁盖谁（28.8） |

不用背。名字里带 `Computed` 的都是布局系统写、你来读的**结算结果**；其余是你写、系统读的**意图**。本章会把上表里的大半逐个请上台。

## 拨一下：把相机撤了

Listing 28-1 里那行 `commands.spawn(Camera2d)` 看着多余——UI 又不在世界里，要相机干什么？把它注释掉，再跑一次。

牌子没了，窗口只剩一片什么都没画过的底灰。再看终端——**日志与正常运行逐字相同**：没有警告，没有报错，一声不吭。

这是本章第一个哑巴坑。UI 树虽然不住在世界坐标里，但它总得知道自己画到**哪块画布**上、画布多大——这份信息是从相机身上拿的（哪台相机、按什么规则拿，28.12 细说）。场上一台相机都没有时，每个节点都找不到归属，整棵 UI 树被静默跳过：不布局、不渲染、不吭声。引擎在这条路径上没有安排任何警告——“挂了 UI 却黑屏、日志干干净净”，先查相机。

牌子挂上了，可它到底占了多大地方？口说无凭，下一节请量尺的进场。
