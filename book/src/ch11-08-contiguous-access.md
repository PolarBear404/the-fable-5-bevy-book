# 连续访问：把整张表借给你

11-3 节把查询快的根源讲成了一句话：同一张 Table 里的组件排成连续一列，遍历就是顺着列走。但普通的 `iter` 只把这份连续性用了一半——它一次递一行，`(name, stock)` 一对一对交到你手上，编译器全程看不出“这些值其实是一整条数组”，想做 SIMD（single instruction, multiple data，一条指令同时处理多份数据）的优化无从下手。存粮年终结算这种“全镇每户都算一遍”的活，一行行递反而碍事。

`contiguous_iter` / `contiguous_iter_mut` 把最后这一半也交出来：**不再一行一行递，一次借出一张表的整列**，每列是一条切片。11-1 节借的是整个 World，这次借的是整张表。


第 1 帧是 11-5 节的旧识——首帧一切皆新，账房全数听见，不必理会。看第 2 帧：迭代只走了**两步**，一步一张表——铺面一张 2 行，素户一张 3 行；每步交来的 `names` 和 `stocks` 是两条等长切片，这张表的整列。D 槽位里能进切片流水线的成员和普通查询大体相同：`Entity` 给 `&[Entity]`，`&T` 给 `&[T]`，`&mut T`、`Ref<T>`、`Option`、`Has` 也各有连续版；`EntityRef` 这类“临场翻档案”的句柄除外——它本来就不是一列数据。

有个细节值得眯眼看：老蔫儿挂着 `Flagged`，组件组合和罗兰、货郎不同，**册都不是同一册，却出现在同一条切片里**。这正是 11-3 节那句“贴记号（SparseSet 组件）不搬表”的回响——连续访问的单位不是册（Archetype）而是**表（Table）**：`Flagged` 不进表，两册档案共用同一张表，整列一起借出。

## 整表盖章

第 2 帧还有桩要案：结算只动了 `stocks[0]`（罗兰一行），账房却听见了**整张表三行**——老蔫儿和货郎的值分毫没变，变更的印章照样盖上了。原因在借条上：11-5 节说印章按行存放、`&mut` 解引用哪行盖哪行，可 `contiguous_iter_mut` 交出的是包着整列的 `ContiguousMut`，对它做 `DerefMut` 拿 `&mut [Stock]` 的那一刻，引擎不知道你接下来要写哪几行——干脆整列全盖。粗，但这正是批量接口的脾气：要逐行精确的印章，就回去用逐行的 `iter_mut`。

不想惊动任何人，出路也和 11-5 节同一个名字：`bypass_change_detection()` 直接给裸切片 `&mut [Stock]`，一个章都不盖。第 3 帧全镇折损 1 袋，账房一个字没听见——数据确实改了（记号户专场随后用普通 iter 读出老蔫儿 6 袋），风险也和 11-5 节说的一样：账实不符，永远无人知晓。

## 两道门槛

不是每个查询都借得到整列，门槛有两道。

**编译期一道**：过滤器必须是“档案级”的——`With`、`Without` 以及它们的元组和 `Or` 组合，这类过滤器只看组件组合、按册匹配（`ArchetypeFilter`）；`Changed`、`Added` 要逐行验印章，整列一把交不出去，编译器直接拒收。

**运行期一道**：查询沾到的组件必须全部是 Table 存储。数据槽或过滤器里混进一个 SparseSet 组件，匹配的行就不再连续，`contiguous_iter_mut` 给 `Err`——第 3 帧的记号户专场撞的正是这堵墙，报错原文把责任人 `With<Flagged>` 点得明明白白：

最后是分寸。切片到手，LLVM 能对循环自动向量化，你也可以亲自上 SIMD——但收益要在数量级上才显出来：几千上万个实体的批量数值运算（物理、粒子、衰减结算）是它的主场，镇上这五户根本用不出差距。日常逻辑，`iter` 一族仍是九成场合的正确答案；这一把和本章其他钥匙一样，认得、会用、少用。
