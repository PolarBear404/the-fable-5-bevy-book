值得回头多看一眼的细节：

- **全场只有一个“写剧本”的系统**。`guild_script` 自始至终只做三类事：`spawn`/`insert`/`remove`/`despawn` 组件、`trigger` 事件——它不认识账房、附魔特效、鉴定师、老板中的任何一位。六位反应者全部是被动挂上去的，这正是第 7 章解耦承诺的推模式版本。
- **第 3 帧的便捷写法**：`commands.entity(目标).trigger(|entity| Identified { entity })`——先锁定实体再补全事件，省去手里攥着 `Entity` 来回传的麻烦。
- **第 5 帧只有钩子出声**。销册是 `Weapon` 的家务事（钩子），没有任何 observer 监听 `Weapon` 的去留——上一节的分工表落了地。
- **第 6 帧的交棒**：observer 的参数表里写着 `MessageWriter<AppExit>`。事件负责“即刻反应”，反应的内容却是投一条消息给 runner 慢慢读——两套机制不是二选一，而是各管一段。

## 怎么选：通知家族全家福

到此为止，“在变化发生时运行逻辑”这件事，你手里已经有五件工具。选型只看两个问题——**多久发生一次**，以及**需要多快反应**：

| 工具 | 模式 | 适合 |
|---|---|---|
| `Message` + `MessageReader`（第 7 章） | 拉：缓冲两帧，读者各自批量消费 | 高频、可攒批的广播：碰撞、计分、输入流 |
| `Event` + `add_observer` | 推：触发即执行 | 低频、要立即反应的全局时刻：存档完成、关卡解锁 |
| `EntityEvent` + `observe`/`add_observer` | 推：瞄准单个实体 | 指名道姓的交互：点击、伤害结算、鉴定 |
| 生命周期事件 `On<Add/Remove/.., T>` | 推：引擎自动触发 | 组件来去引发的联动：装备特效、缓存失效 |
| 组件钩子 `#[component(on_add = ..)]` | 推：组件固有行为 | 组件自身的不变式：索引维护、配对登记 |

一条经验性的分界线：**每帧都在大量发生的事，用 Message 攒着批处理；偶尔发生、发生了就要立刻办的事，用 Event 和 Observer**。拿不准时先想想第 7 章 DJ 的台词——“有没有”比“有几条”重要的场合，往往说明你要的其实是缓冲。

## 小结

- **Event 是即时事件**：`#[derive(Event)]` 定义、无需注册，`commands.trigger` 走命令队列，应用的那一刻所有匹配 observer 当场同步跑完；没有缓冲、没有游标，错过即不存在
- **Observer 是头号参数为 `On<E>` 的系统**：其余参数随意声明；同一事件可挂任意多个，但相互顺序无保证；observer 的真身是挂着 `Observer` 组件的实体
- **EntityEvent 带准星**：`entity` 字段（或 `#[event_target]`）锁定目标，全局 observer 与 `.observe()` 挂的专属 observer 都会运行；目标可能已销毁，`get` + `let-else` 防御
- **生命周期事件五件套**：`Add`/`Insert` 管上身（首次/每次），`Replace`/`Remove` 管清退（值/有无），`Despawn` 管销毁；移除类事件运行在数据消失之前；纯改值不触发任何一个
- **observer 触发链在同一个同步点内递归跑完**：联动零延迟，但引擎不查环，无限连锁会栈溢出
- **组件钩子是组件的构造/析构函数**：`fn(DeferredWorld, HookContext)` 固定签名，每组件每种至多一个；添加时钩子先于 observer，移除时 observer 先于钩子

1. **零改动加戏**：给 Listing 8-3 的诅咒之剑再挂第二个专属 observer（比如“诅咒之剑：还有完没完！”），同时给铁剑也配一句专属台词。运行验证：同一实体可以叠多个专属 observer，且原有代码一行不动。
2. **顺序侦探**：把 Listing 8-6 剧本里的第 3 帧（拆附魔）整段删掉，先在纸上预测第 4、5 帧各会打出哪几行、顺序如何，再运行对答案。提示：第 4 帧的 `Add` 还响吗？
