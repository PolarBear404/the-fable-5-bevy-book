# 单位阅兵

`Node` 里凡是量长短的字段——宽高、边距、内距、格子尺寸——类型都是 **`Val`**。它是个枚举，七种量法：

- **`Auto`**——不表态，让布局自己定。多数字段的默认值，具体“自己定”成什么样，随字段而异：宽高上是“按内容或弹性规则来”，边距上是“归零”（28.7 会见到它的一手绝活）；
- **`Px(f32)`**——逻辑像素，写多少是多少；
- **`Percent(f32)`**——父节点的百分之几。参照谁，下面有细则；
- **`Vw(f32)` / `Vh(f32)`**——视口宽度/高度的百分之几。视口（viewport）就是 UI 所在画布的可见范围，通常是窗口；
- **`VMin(f32)` / `VMax(f32)`**——视口窄边/宽边的百分之几，横竖屏通吃。

第 16 章 `FontSize` 的那套响应式单位，正是从这里借的形。全书代码里你还会常看到小写的 `px(320)`、`percent(50)`、`vw(2)`——它们是同名便捷函数，`px(320)` 就是 `Val::Px(320.0)`，官方示例的主流写法，省得满屏 `Val::`。

三个“50%”报了三个数，这正是要点：

- **`Vw(50)` → 640**：视口宽 1280 的一半，分毫不差。视口单位量的是画布本身，不理会父节点是谁、留了多少衬；
- **`Percent(50)` → 624**：百分比量的是**父节点**。父节点（校场）宽 1280，但四边各有 16 像素 padding，能给子女住的**内容区**只剩 1280 − 32 = 1248，一半 624。差的这 16 像素就是两种单位的分水岭——`Percent` 认爹，`Vw` 认天；
- **`Vh(50)`、`VMin(50)` → 360**：视口高 720 的一半。眼下窗口横着放，窄边就是高，`VMin` 与 `Vh` 报数相同——等会儿拖个瘦高窗就分家了。

## 谁参照谁

`Percent` 的参照物按字段分三路，规矩全盘继承自 CSS：

| 字段 | `Percent` 参照 |
|---|---|
| `width`、`min_width`、`max_width` | 父内容区的**宽** |
| `height`、`min_height`、`max_height` | 父内容区的**高** |
| `left`、`right` | 父内容区的宽 |
| `top`、`bottom` | 父内容区的高 |
| `margin`、`padding`、`border` | 一律参照父内容区的**宽**——连上下方向的也参照宽 |

最后一行是 CSS 沿袭下来的著名怪癖：`padding: UiRect::all(percent(10))` 四边**一样厚**（都是父宽的 10%），而不是“上下参照高、左右参照宽”。知道有这条就够，踩到时不至于怀疑人生。（还有半条备着 28.7 用：`left` 等四个字段用在**绝对定位**上时，参照的是父级的 padding box——比内容区多出衬边那一圈。）

## 拨一下：拖个瘦高窗

拖动窗口边框，把它拽成一个瘦高个（客户区约 547×684 逻辑像素），再按空格：

```text
报数——
      Px(300) 宽 300 逻辑像素
  Percent(50) 宽 257 逻辑像素
       Vw(50) 宽 273 逻辑像素
       Vh(50) 宽 342 逻辑像素
     VMin(50) 宽 273 逻辑像素
```

`Px` 纹丝不动，其余全体换算：`Vw` 跟着新窗宽走（547 的一半），`Vh` 跟着新窗高（684 的一半），`Percent` 还是父内容区的一半（547 − 32 后取半）。最有意思的是 **`VMin` 换了跟班**——窗口竖起来之后窄边从“高”变成了“宽”，它立刻改报 273，跟 `Vw` 并了队。做“无论横竖屏都占安全比例”的元素（头像、方形按钮），`VMin` 是首选刻度。

## 拨一下：UiScale 这把全局旋钮

还有一把跟单位纠缠的旋钮要当场试清楚：资源 **`UiScale`**（一个 newtype，`UiScale(f32)`，默认 1.0），号称“整个 UI 的全局缩放”。按 U 把它拨到 1.5：

```text
水牌师傅：UiScale 拨到 1.5。
报数——
      Px(300) 宽 300 逻辑像素
  Percent(50) 宽 411 逻辑像素
       Vw(50) 宽 427 逻辑像素
       Vh(50) 宽 240 逻辑像素
     VMin(50) 宽 240 逻辑像素
```

看画面：`Px` 那条牌**肉眼可见地长大了一半**，其余四条的实际尺寸几乎没动。可报数恰好反过来——`Px` 报数没变，其余全变了。两边合起来才是 `UiScale` 的真面目：

**它只放大 `Val::Px` 这一种值**。拨到 1.5 的意思是“一个 UI 逻辑像素画成 1.5 倍大”，于是写 `px(300)` 的牌子画出去宽了一半，账面还是 300；而视口的逻辑尺寸被这把旋钮除小了（1280 ÷ 1.5 ≈ 853），`Vw(50)` 的账面从 640 缩到 427——但换算回屏幕，427 × 1.5 = 640，**物理占比原地不动**。百分比、视口单位本来就是比例，比例不吃缩放。

这正是它的用途所在：设置菜单里那档“界面大小”，一个资源拨过去，所有用 `Px` 写的 HUD 齐步变大，占比式布局照旧撑满——两不相扰。跟第 16 章 `RemSize` 的“大字模式”是同一路数，一个管字号，一个管 UI 尺寸。
