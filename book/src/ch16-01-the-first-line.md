**`Text2d`**（2D 世界文本）是个普通组件，里面就装着一个 `String`。它的全部排场来自 required components（第 3 章的规矩）——挂上它，引擎自动补齐一整套：

- `TextFont`——用哪副字模、刻多大字（这一节先用默认值）；
- `TextColor`——字色，默认白；
- `TextLayout`——对齐与换行的规矩（16.8 节）；
- `TextBounds`——文字的地界，默认无界（16.8 节）；
- `LineHeight` 与 `LetterSpacing`——行高与字距，各是一个独立组件（16.6 节）；
- `Anchor`——整块字钉在 `Transform` 的哪个点上，**与 Sprite 共用同一个锚点组件**（第 15 章），默认 `CENTER`；
- 还有 `Transform`、`Visibility` 这些 2D 实体的标配。

所以 `Text2d` 的世界观和 `Sprite` 完全一致：它就是一张“画着字的图”，能摆放、能旋转、能缩放、能配锚点，跟着第 12 章的变换体系走。机器试通了，老雷把秋白的手稿递过来。词换上去：

英文行好好的，秋白的十个字上台变成了十个**空心方框**。控制台倒是开了腔——启动日志中间冒出两行没头没脑的抱怨：

```text
ICU4X data error: No segmentation model for language: ja
ICU4X data error: No segmentation model for language: ja
```

先说结论：**这两行与豆腐块无关**，而且无害。它是排版引擎里的**分词器**在说话：排中日韩文本时，它想加载一个“词边界”模型（分词服务于双击选词、按词跳光标这类文本编辑功能），引擎内置的排版库没有带这个模型，于是每逢新的中日韩文本进排版机，它就抱怨一声——条数不定，内容永远是这一句。它**不影响渲染，也不影响换行**：断行走的是另一套规则（16.8 节能看到中文换行毫不费力）。这声抱怨也压不掉——注意它没有时间戳、没有日志级别，和上下的 Bevy 日志长得完全不同：它绕过了 Bevy 的日志系统直落控制台，`RUST_LOG` 管不着它。本章后面的示例它都会在场，把它当背景音就好。

> **真正的坏消息是没有消息**：豆腐块本身——引擎明明缺了十个字形——却是零报错、零警告、零日志。控制台里唯一开口的那两行，说的还是另一件事。把“有输出”当成“这就是问题所在”，是排查这类现象时最容易走进的岔路。

豆腐块的来历：字体文件里每个字符对应一个**字形**（glyph，字的轮廓数据）；文本引擎排版时拿着字符去字体里查字形，查不到就用 0 号字形顶替——字体设计师管它叫 `.notdef`（not defined），通常画成一个空框。也就是说，**豆腐块是字体在喊“我没有这个字”**。

为什么默认字体没有中文？Bevy 内置的默认字体是 Fira Mono 的一个子集，总共只有 **95 个字形**——ASCII 可打印区，从空格到 `~`，一个不多。这不是疏忽：一副像样的中文字体动辄一两千万字节，内置进引擎谁都得背着。更要紧的是第二件事：

> **Bevy 默认不看你的系统字体。** 桌面程序里“缺字回退到系统字体”是操作系统的服务；Bevy 的文本引擎默认只在**已加载的字体资产**里找字形，不会扫描 Windows 的字体目录。所以“我电脑上装着微软雅黑”帮不上忙——引擎压根不看。确实有一扇门能让它看（16.4 节），但那是一个要权衡的发行决策，不是默认行为。

两条线索拼起来，路只有一条：把一副有中文字形的字体当成**资产**交进库房。这正是下一节的事。
